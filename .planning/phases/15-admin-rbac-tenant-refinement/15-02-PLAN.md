## Design detail

### Platform authorization

- Routes hiện `@Roles(ShopRole.super_admin)` **không dùng tenant**: đổi sang kiểm tra `user.platform_role === platform_super` HOẶC decorator mới `@PlatformRoles(...)` để tránh nhầm với `ShopRole`.  
- **Legacy window (chọn một, ghi trong code comment):** trong N phiên bản, vẫn cho phép user có `UserShopRole.role === super_admin` nếu `platform_role` null — log deprecation; hoặc strict: sau migration luôn require `platform_role`. **Khuyến nghị:** strict sau backfill vì migration 15-01 đã set platform.

### Tenant authorization

- `shop_staff`:  
  - GET-safe endpoints: allow where today `shop_admin` is allowed **chỉ** if method GET/HEAD hoặc explicit allowlist per controller.  
  - POST/PATCH/DELETE: deny với 403 trừ danh sách ngoại lệ rõ (nếu không có ngoại lệ, staff read-only hoàn toàn).  
- Cách triển khai gợi ý: helper `assertShopWriteRole(user, tenantId)` hoặc metadata decorator `@ShopRoles({ read: [...], write: [...] })` — chọn một pattern và dùng nhất quán trong phase.

### Shops service

- `addShopAdmin` / tương đương: nhận `role` trong DTO (`shop_admin` \| `shop_staff` \| giữ tương lai); validate enum; audit log với action mới.  
- Không cho phép gán `platform_role` qua API tenant thường (chỉ platform route hoặc script).

### JWT / session

- Nếu JWT chứa shop_roles snapshot: refresh hoặc TTL — khi đổi role DB, đảm bảo client gọi refresh hoặc re-login; document trong auth flow.
