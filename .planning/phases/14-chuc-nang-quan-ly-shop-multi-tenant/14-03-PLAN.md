# Phase 14 Plan 03: Admin tenant selection UI và super-admin shop management

## Objective
- **What:** Frontend UI để user chọn shop active, indicator hiện trong Layout, và trang ShopsPage cho super admin CRUD shops.
- **Why:** MULT-02 — admin phải rõ ràng biết mình đang thao tác trên shop nào.
- **Output:** `ShopContext`, `ShopSelector` dropdown, Layout header indicator, `ShopsPage`.

## Context
@.planning/REQUIREMENTS.md
@.planning/phases/14-chuc-nang-quan-ly-shop-multi-tenant/14-CONTEXT.md

## Tasks

<task type="code">
  <name>Tạo ShopContext để quản lý active shop state</name>
  <files>frontend/src/context/ShopContext.tsx</files>
  <action>
    Tạo `ShopContext` React context với:
    - `activeShop`: Shop | null
    - `allowedShops`: Shop[]
    - `switchShop(shopId: string): Promise&lt;void&gt;` — gọi `POST /auth/switch-shop`, update token trong storage, re-fetch user context.
    Khi token mới về, decode payload để lấy `active_shop_id` và cập nhật `activeShop`.
    Nếu user load lại trang → đọc token cũ → restore activeShop từ payload.
    Nếu `allowedShops.length === 0` → clear session và redirect về `/login`.
  </action>
  <verify>Unit test ShopContext: switchShop gọi đúng API, token được update, activeShop thay đổi.</verify>
  <done>Context quản lý đúng shop state qua token lifecycle.</done>
</task>

<task type="code">
  <name>Cập nhật axios interceptor để attach JWT mới sau switch-shop</name>
  <files>frontend/src/api/axios.ts</files>
  <action>
    Đảm bảo axios interceptor luôn đọc token từ storage mới nhất trước mỗi request.
    Sau `switchShop()`, token mới được ghi vào storage → interceptor tự động dùng token mới ở request kế tiếp.
    Không cần thêm `x-tenant-id` header — JWT đã có `active_shop_id`.
  </action>
  <verify>Manual test: switch shop → gọi GET /items → check header Authorization trong Network tab có bearer token mới.</verify>
  <done>Axios luôn dùng đúng token sau switch.</done>
</task>
